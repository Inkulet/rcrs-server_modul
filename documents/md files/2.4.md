# **Разработка архитектуры своего решения**

На основе выявленных архитектурных ограничений существующих систем, для реализации в рамках данного проекта выбирается подход на основе адаптивной мультифакторной функции полезности.

Данный метод проектируется как улучшение сильных сторон аналога AIT-Rescue Strategy. Он обеспечит:

1. Вычислительную сложность O(M), гарантируя время отклика в пределах 50–100 мс (Критерий 1).

2. Полную коммуникационную автономность (неявная координация), резервируя радиоканалы исключительно для сенсорных данных среды (Критерий 2).

3. Интеграцию компонента срочности furgency, явно учитывающего таймеры жизни (TTL) и стадии возгорания зданий (Критерий 3).

4. Внедрение локально вычисляемого социального фактора fsocial(Ni), который реализует механизм децентрализованного предотвращения интерференции, снижая приоритет задачи пропорционально наблюдаемой плотности агентов-союзников вокруг нее (Критерий 4).

Предложенная архитектура позволит агентам децентрализованно расходиться по карте, динамически балансируя между выполнением наиболее приоритетных задач и предотвращением дублирования усилий, что является необходимым условием для повышения глобального показателя эффективности системы в симуляторе RCRS.

1. # **Use-case диаграмма** 

Для формализации функциональных требований к разрабатываемому модулю распределения задач проведен анализ прецедентов (Use-Case). В архитектуре симулятора RCRS взаимодействие строится вокруг концепции дискретных тактов.

Система взаимодействует с двумя основными типами акторов:

* Ядро симулятора (RCRS Kernel) \- внешний системный актор. Отвечает за симуляцию физических процессов и рассылку сенсорных данных. Ядро является инициатором каждого цикла работы агента и получателем его итоговых команд.  
* Агент-спасатель (Rescue Agent) \- главный актор системы, представляющий автономную интеллектуальную единицу. Ввиду гетерогенности команды, данный актор выступает базовым классом и наследуется тремя специализированными ролями: Агент пожарной охраны (FireTeam), Агент полиции (Police Force), Агент скорой помощи (Ambulance Team).

## Описание ключевых сценариев

Процесс принятия решения за один такт представляет собой строгий вычислительный конвейер. Обязательная последовательность выполнения шагов описывается следующими прецедентами:

* UC-1: Обновить модель мира. Ядро симулятора отправляет агенту пакет визуальных данных. Агент инициирует этот прецедент для слияния новых параметров (HP, Temperature, Damage) с историческими данными в локальной памяти.  
* UC-2: Фильтровать доступные цели. Базовый этап отсеивания нерелевантных задач до применения математических вычислений. Исключение нежизнеспособных целей. Сценарий активируется (расширяет UC-2), если параметры цели критические: HP=0, здание имеет стадию горения (4,...,8), либо расчетное время спасения превышает оставшееся время жизни TTL  Ttravel \+ Twork.  
* UC-3: Оценить полезность задач. Вычисление математической функции Uij со сложностью O(M). Сценарий инкапсулирует вычисление базовых метрик (срочность, дистанция, трудоемкость) и включает в себя под-сценарий расчета социального фактора (UC-4).  
* UC-4: Учесть состояние союзников. Обязательный под-сценарий (связь \<\<include\>\> от UC-3). Агент анализирует вектор состояний известных союзников A(t) и математически снижает приоритет переполненных задач без использования радиосвязи, обеспечивая выполнение Критерия 4\.  
* UC-5: Выбрать целевую задачу. Агент производит выбор оптимальной цели (tbest), максимизирующей функцию полезности, с учетом заданного коэффициента гистерезиса для предотвращения цикличного переключения между задачами.  
* UC-6: Построить маршрут к цели. Инициация алгоритма A\* на графе G \= (V, E) для построения маршрута к выбранной цели tbest.  
* UC-7: Отправить команду. Формируется и отправляется итоговая команда Ядру симулятора.  
* UC-8: Запросить пополнение воды. Прецедент специфичен для роли FireTeam. Инициализируется при исчерпании воды, агент формирует команду движения к ближайшему источнику водоснабжения вместо выполнения стандартного цикла выбора задачи.

2. # **Диаграмма компонентов**

Архитектура разбита на слабо связанные модули, выстроенные в строгий вычислительный конвейер.

## **Ключевые подсистемы:**

* Слой восприятия. Отвечает за прием сырых сенсорных данных от ядра RCRS по протоколу TCP/IP, разрешение конфликтов версионирования информации и обновление разделяемой In-Memory модели мира.  
* Слой данных. В связи с жесткими ограничениями на время отклика, использование классических СУБД исключено. Использование in-memory СУБД, также нецелесообразно: структура данных симуляции представляет собой взвешенный граф G \= (V, E) с объектами-агентами A(t) и задачами T(t), операции над которым выполняются эффективнее через прямую адресацию объектов в памяти, чем через слой трансляции запросов. В связи с этим архитектура базируется на специализированной In-Memory структуре данных \- объектно-ориентированном кэше модели мира, хранящем граф дорожной сети, векторы состояний агентов и задач непосредственно в куче JVM/Python-процесса агента и обновляемом каждый такт на основе сообщений от сервера симуляци. Слой представляет собой объектно-ориентированный кэш в оперативной памяти, хранящий графовую модель дорожной сети G \= (V, E), вектор состояний агентов A(t) и активных задач T(t).  
* Слой фильтрации. Изолированный компонент, реализующий эвристические правила отсечения нерелевантных задач до начала математических расчетов.  
* Модуль вычисления функции полезности. Центральный вычислительный узел с асимптотической сложностью O(M). Делегирует расчеты специализированным подкомпонентам: вычисление фактора срочности, геометрического расстояния, трудоемкости и социального фактора. Изоляция этих формул в отдельные компоненты позволяет легко адаптировать весовые коэффициенты (w) функции.  
* Слой выбора цели и навигации. Принимает матрицу полезности от предыдущего слоя, применяет защиту от гистерезиса и инициирует алгоритм поиска кратчайшего пути на графе дорог к итоговой цели.  
* Слой исполнения. Формирует конечный пакет команды в зависимости от типа агента и отправляет его обратно в ядро симулятора. Разработанная диаграмма компонентов представлена на рисунке 2\.
